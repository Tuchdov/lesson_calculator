## Problem Statement

Port the existing Python/Streamlit `vocal_business_calc` app to a React SPA (Vite + React 19) in the `lesson_calculator` repo. The app is a personal tool for a vocal teacher to calculate monthly lesson payments from Google Calendar events. The Python version is fully functional. The goal is a JS+React version deployed to Vercel.

## What Makes This Cool

- The business logic (Hebrew/English student name parsing, regularity detection, duration rounding) is self-contained — it ports to a zero-dependency JS module with no runtime libraries
- Moving from Python/Streamlit server-side to a pure SPA eliminates the server maintenance burden entirely
- Google Identity Services (GIS) replaces the custom HMAC OAuth state validation — cleaner security model for browser apps with no client secret exposed
- Vercel deployment means zero ops: push to main, it deploys
- Deno's built-in test runner covers the lib layer with no test framework overhead

## Constraints

- Feature parity with Python app: Calculator page + Custom Prices page
- Single user (the teacher's own Google account)
- Vercel deployment target (static SPA)
- Deno used as package manager and test runner
- No Node.js backend server

## Premises

1. Feature parity is the goal — no new features in the initial port
2. Pure SPA using Google Identity Services (GIS) browser-based OAuth — no backend needed
3. User settings (custom prices) stored in `localStorage`, keyed by SHA-256 of email; GIS access token stored in `sessionStorage` (expires with tab) — do not swap these
4. Business logic ports to vanilla JS ES module with zero runtime dependencies
5. Tests use Deno's built-in test runner (`deno test`) with `jsr:@std/assert`; tests only import from `src/lib/` and `src/api/` — never from `src/components/` or `src/hooks/` (avoids JSX/browser API issues in Deno)
6. Deploy to Vercel via `vite build` — `VITE_GOOGLE_CLIENT_ID` set as Vercel env var

## Approaches Considered

### Approach A: Flat SPA (minimal)
- Summary: App.jsx handles everything, 3-4 files total, no routing, read-only results table
- Effort: S | Risk: Low
- Pros: Fastest to ship, fewest moving parts
- Cons: Cuts the editable per-student price override table — a key feature of the Python app

### Approach B: Full-featured SPA with React Router (ideal)
- Summary: React Router for Calculator/Custom Prices, custom hooks, TanStack Table, full test suite
- Effort: M-L | Risk: Low-Med
- Pros: Most scalable, URL-based navigation
- Cons: React Router adds a dependency; TanStack Table is overkill for 10 columns

### Approach C: Middle-ground SPA (chosen)
- Summary: State-based page switching (no router), controlled HTML table for price editing, lib modules, Deno tests
- Effort: M | Risk: Low
- Pros: Mirrors Python app UX exactly, no third-party table library, idiomatic React, Deno test fits naturally
- Cons: No URL-based navigation between pages (acceptable for a personal single-user tool)

## Recommended Approach

**Approach C: Middle-ground SPA**

Mirrors the Python app's page-switching model (sidebar radio button = React state). No third-party table library — the editable table is a controlled `<table>` with `<input>` elements, which is simpler than the Streamlit `data_editor` and requires no additional deps. Deno test runs the lib layer tests with zero config. Vercel deploys from `dist/` after `vite build`.

## Google Identity Services (GIS) Auth Design

GIS replaces the custom HMAC state validation in `web_auth.py`. The browser-side flow:

1. Load `https://accounts.google.com/gsi/client` script tag in `index.html`; if script fails to load (ad blockers, network), `AuthScreen.jsx` shows: "Google sign-in unavailable — check your network or browser extensions"
2. `useAuth.js`: call `google.accounts.oauth2.initTokenClient({ client_id, scope, callback })` — this is the **implicit token flow** (not PKCE; PKCE is for auth code flow). GIS handles state internally. No client secret is needed or used.
3. On sign-in button click: `tokenClient.requestAccessToken()`
4. GIS callback receives `{ access_token, expires_in, token_type }`
5. Fetch `https://www.googleapis.com/oauth2/v1/userinfo` with access token to get name/email/picture
6. Store token + user info in `sessionStorage` (expires with tab close, ~1 hour max)
7. Token expiry UX: when Calendar API returns 401, try silent re-auth via `requestAccessToken({ prompt: '' })` first; if that fails, show a "Session expired — sign in again" banner

Required: `VITE_GOOGLE_CLIENT_ID` env var (safe in browser — public client, no secret)
Required: Google Cloud Console → Authorized JavaScript origins = `http://localhost:5173` (dev) + Vercel URL (prod)

## Business Logic Port Map

| Python function | JS function | Notes |
|---|---|---|
| `lesson_duration_minutes()` | `lessonDurationMinutes()` | Same fuzzy rounding (27-33→30, 42-48→45, 57-63→60) |
| `is_target_lesson()` | `isTargetLesson()` | Same Hebrew phrase check (`פיתוח קול`) |
| `is_cancelled_lesson()` | `isCancelledLesson()` | `String.prototype.toLowerCase()` replaces `casefold()` |
| `is_paid_cancellation()` | `isPaidCancellation()` | Same Hebrew phrase (`ביטול בתשלום`) |
| `infer_student_name()` | `inferStudentName()` | Same separators ` - `, ` | `, `: ` + normalize |
| `normalize_student_name()` | `normalizeStudentName()` | Strip parenthetical tokens, take first 2 words |
| `is_regular()` | `isRegular()` | Same 4-lesson / 7-day-window logic |
| `resolve_price_table()` | `resolvePriceTable()` | Same legacy flat + tiered format support |
| `calculate_payments()` | `calculatePayments()` | Same aggregation, returns `{rows, summary}` |
| `month_window()` | `monthWindow()` | `Date` instead of `datetime` |
| `to_dt()` | — | `new Date(str)` parses ISO 8601 natively |

**Python `dateutil` → JS built-in `Date`**: JS `new Date("2026-02-01T10:00:00+02:00")` correctly parses timezone-aware ISO 8601 strings. No external library needed.

**Casefold → lowercase**: Hebrew `casefold()` in Python is equivalent to `toLowerCase()` in JS for the specific keywords used.

**`resolvePriceTable()` format support** — two custom price formats, checked in order:
```json
// Tiered format (new): per-regularity custom prices
{ "Alice": { "regular": {"60": 200, "45": 150, "30": 120}, "non_regular": {"60": 250, "45": 150, "30": 150} } }

// Legacy flat format: single price table for both types
{ "Alice": {"60": 300, "45": 250, "30": 220} }
```
Detection: if entry has `"regular"` or `"non_regular"` keys → tiered. If entry has `"60"`, `"45"`, `"30"` keys → legacy flat.

**`inferStudentName()` separator priority** — tried in order, first match wins:
0. If `student_name_regex` config field is set: try regex first (first capture group); if it captures past `פיתוח קול`, trim the phrase and normalize
1. ` - ` (space-dash-space)
2. ` | ` (space-pipe-space)
3. `: ` (colon-space)
4. If none match and title contains `פיתוח קול`: extract everything before the phrase → normalize (strip parentheticals, take first 2 tokens)
5. Fallback: normalize the entire title

The `student_name_regex` in `pricing_config.json` (default: `^([^-|:]+)`) is the optional regex — it's tried first and the separator logic is the fallback, not an alternative approach.

**`fetchEvents()` pagination**: Google Calendar API returns `nextPageToken` in the response when there are more events. `calendarApi.js` loops: `while (nextPageToken) { fetch(params + pageToken); }`. Page size defaults to Google's default (250 events per page). Termination: when `nextPageToken` is absent from response.

**Default month picker value**: current year + current month (same as Python app's `datetime.now()` default).

**`EditableTable.jsx` scope**: used ONLY by `CalculatorPage.jsx` for the per-session price overrides. `CustomPricesPage.jsx` uses a plain `<table>` + `<input>` inline, not `EditableTable` — they are different UX patterns (session-level overrides vs. persistent custom defaults).

## User Settings Design

Replace server-side `user_data/{sha256-email}.json` with `localStorage`:
- Key: `vocalCalc:settings:{sha256hex(email)}` where SHA-256 is computed via `crypto.subtle.digest` (async Web Crypto API)
- Value: same JSON structure as Python: `{ custom_prices: { "student name": { regular: {...}, non_regular: {...} } } }`
- On load: read from localStorage, parse JSON, default to `{ custom_prices: {} }` if missing
- On save: serialize to JSON, write to localStorage
- Existing server-side JSON files are NOT migrated — clean start

**Async key derivation in `useUserSettings.js`:** SHA-256 of email is computed once on mount after auth resolves. The hook exposes `{ settings, saveSettings, isReady }`. Components wait on `isReady` before reading/writing (prevents reading from the wrong key before the hash is computed). Deno tests mock `localStorage` using a `Map`-backed stub and call `sha256Hex()` directly — no browser globals needed.

**`localStorage` in Deno tests:** `localStorage` is a browser API not available in Deno. `userSettings.js` must accept an optional `storage` parameter (default: `globalThis.localStorage`), allowing tests to inject a `Map`-backed stub:
```js
const stub = { getItem: (k) => map.get(k), setItem: (k, v) => map.set(k, v) };
loadUserSettings(email, stub); // testable without browser globals
```

## `pricing_config.json` Schema

Bundled as `public/pricing_config.json` (served statically by Vite). Schema matches the existing Python app's config exactly:

```json
{
  "calendar_id": "primary",
  "cancelled_keywords": ["cancelled", "canceled", "cancel", "בוטל", "מבוטל"],
  "student_name_regex": "^([^-|:]+)",
  "prices": {
    "regular":     { "60": 250, "45": 200, "30": 180 },
    "non_regular": { "60": 270, "45": 220, "30": 200 }
  },
  "custom_prices": {}
}
```

Fields consumed by `calendarPayments.js`:
- `calendar_id`: passed to Calendar API `calendarId` parameter
- `cancelled_keywords`: list of strings, checked case-insensitively against event titles
- `student_name_regex`: optional regex to extract student name (first capture group)
- `prices.regular` / `prices.non_regular`: default price tables by lesson duration
- `custom_prices`: per-student overrides (populated at runtime from localStorage, not from this file)

`custom_prices` in the bundled file is always `{}` — runtime custom prices come from `useUserSettings.js`.

## Hook API Surfaces

**`useAuth()`** returns `{ userInfo, isAuthenticated, signIn, signOut }`
- `userInfo`: `{ email, name, picture }` (null when not authenticated)
- `isAuthenticated`: boolean
- `signIn()`: calls GIS `requestAccessToken()` (no-op if already authenticated)
- `signOut()`: clears sessionStorage, resets state

**`useCalendarData(config)`** returns `{ rows, isLoading, error, calculate }`
- `rows`: array of `{ student, student_type, lessons_60, lessons_45, lessons_30 }`
- `isLoading`: boolean — shown as spinner in CalculatorPage
- `error`: string | null — shown as error banner
- `calculate(monthStr)`: triggers fetch + calculation, updates rows; on re-run with a new month, rows reset to the new calculation's defaults (overrides discarded — same behavior as Python app)

**`useUserSettings(userEmail)`** returns `{ settings, saveSettings, isReady }`
- `settings`: `{ custom_prices: { ... } }`
- `isReady`: false until SHA-256 key is derived (async)
- `saveSettings(newSettings)`: writes to localStorage

Note: `crypto.subtle` is only available in secure contexts (HTTPS or localhost). The app requires HTTPS in production and staging. Do not use plain HTTP staging URLs.
